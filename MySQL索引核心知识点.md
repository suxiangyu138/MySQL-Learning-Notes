# MySQL 索引核心知识点

## 一、索引是什么
1. **定义**：索引是**排好序的快速查找数据结构**，作用是**加速查询**，类似书的目录。
2. **核心本质**：用**空间换时间**，提升查询速度，会增加写入/更新开销。
3. **默认存储引擎**：InnoDB，索引底层数据结构是 **B+ 树**。

## 二、B+ 树索引（InnoDB 核心）
### 1. 为什么用 B+ 树？
- 多路平衡，**磁盘 IO 次数极少**（查询效率稳定）
- 所有**数据存在叶子节点**，非叶子节点只存索引键
- 叶子节点**双向链表相连**，支持**范围查询、排序、分组**

### 2. 两种索引结构（最关键）
1. **聚簇索引（Clustered Index）**
    - 主键索引，**叶子节点存整行数据**
    - 一张表**只有一个**聚簇索引
    - 建议：表必须建主键（推荐自增 int/bigint），避免页分裂

2. **二级索引（普通索引）**
    - 非主键索引，**叶子节点存主键值**
    - 查询时先查二级索引拿到主键，再回聚簇索引查数据 = **回表**

## 三、索引类型（开发常用）
1. **主键索引**：`PRIMARY KEY`，唯一且非空，聚簇索引
2. **唯一索引**：`UNIQUE`，值唯一，允许 NULL
3. **普通索引**：`INDEX`，最常用，仅加速查询
4. **联合索引**：`INDEX(a,b,c)`，**最左前缀原则**核心
5. **覆盖索引**：查询的列**全部在索引里**，无需回表，性能极高

## 四、联合索引 & 最左前缀原则
### 1. 联合索引规则
`INDEX(a, b, c)` 等效创建了三个索引：
- `(a)`
- `(a,b)`
- `(a,b,c)`

### 2. 最左前缀原则
- **必须从最左侧列开始匹配**，跳过左边列索引失效
- 范围查询（`> < between like 前缀%`）之后的列索引失效

✅ 有效：
`where a=1`
`where a=1 and b=2`
`where a=1 and b=2 and c=3`
`where a=1 order by b`

❌ 失效：
`where b=2`（跳过 a）
`where a=1 and c=3`（跳过 b）
`where a=1 and b>2 and c=3`（b 范围后 c 失效）

## 五、索引最佳实践（必背）
1. **最左前缀**：联合索引按查询频率、区分度从左到右建
2. **区分度高的放前面**：性别（区分度低）不适合建索引
3. **避免索引失效**
    - 索引列上运算、函数、类型转换
    - `!=` / `is not null` / `not like`
    - `like %xx` 前缀模糊匹配
4. **优先覆盖索引**：查询只查需要的列，避免 `select *`
5. **控制索引数量**：单表索引建议 **≤5 个**，写入会变慢
6. **主键推荐**：自增整数（int/bigint），避免 UUID 导致页分裂

## 六、索引失效场景（高频）
1. 索引列使用函数：`where func(col)=xx`
2. 索引列运算：`where col+1=xx`
3. 类型隐式转换：字符串不加引号
4. 违反最左前缀
5. `or` 连接非索引列
6. 模糊查询 `like %xx`

## 七、执行计划 EXPLAIN（排查索引神器）
看这 3 个字段就行：
1. **type**：访问类型
   `system > const > eq_ref > ref > range > index > ALL`
   **range 及以上才算有效索引**，ALL 是全表扫描（必须优化）
2. **key**：实际使用的索引
3. **Extra**
    - `Using index`：覆盖索引（最优）
    - `Using filesort`：文件排序（需优化）
    - `Using temporary`：临时表（严重性能问题）

## 八、索引优缺点
### 优点
- 大幅提升查询、排序、分组速度
- 降低数据库扫描的数据量

### 缺点
- 占用磁盘空间
- **增删改变慢**（需要维护索引）
- 过多索引会导致优化器选择困难

---

### 总结
1. InnoDB 索引是 **B+ 树**，主键是聚簇索引，二级索引存主键
2. 联合索引核心：**最左前缀原则**
3. 开发口诀：**建联合索引、避免索引失效、多用覆盖索引**
4. 排查用 `EXPLAIN`，看 `type` 和 `key` 即可判断索引是否生效
