# MySQL 面试高频题+精简标准答案

## 一、存储引擎 & 基础
### 1. InnoDB 和 MyISAM 区别
1. InnoDB：支持**事务、行级锁、MVCC、外键**，崩溃安全恢复，适合读写混合、高并发业务。
2. MyISAM：不支持事务、行锁，只有表锁，查询性能高，崩溃易丢数据，现在基本淘汰。

### 2. 事务四大特性 ACID
- **A原子性**：事务要么全部执行，要么全部回滚。
- **C一致性**：事务前后数据完整性约束不变。
- **I隔离性**：多个事务之间互相隔离互不干扰。
- **D持久性**：事务提交后数据永久落地，崩溃不丢失。

### 3. 四大隔离级别 & 问题
由低到高：
1. 读未提交：存在**脏读、不可重复读、幻读**。
2. 读已提交(RC)：解决脏读，存在不可重复读、幻读。
3. **可重复读(RR，MySQL默认)**：解决脏读、不可重复读，存在幻读；InnoDB通过间隙锁抑制大部分幻读。
4. 串行化：全部问题解决，完全串行，并发极低。

### 4. 脏读、不可重复读、幻读
- 脏读：一个事务读到另一个事务**未提交**的数据。
- 不可重复读：同一事务内，两次查同一数据，结果不同（别的事务修改提交）。
- 幻读：同一事务范围查询，前后行数不一致（别的事务插入/删除）。

## 二、索引（重中之重）
### 1. 为什么索引用 B+ 树
1. 多路平衡树，层高更低，磁盘IO少。
2. 非叶子节点只存索引键，内存利用率高。
3. 叶子节点双向链表，**范围查询、排序、分页**极强。
4. B树叶子非链表，范围查询弱；哈希不支持范围与排序。

### 2. 聚簇索引 & 二级索引
- 聚簇索引：**主键索引**，叶子节点存整行数据，一张表只有一个。
- 二级索引（普通/联合/唯一）：叶子只存**主键值**，查到后需要回表拿完整数据。

### 3. 回表 & 覆盖索引
- 回表：通过二级索引拿到主键，再走聚簇索引查整行数据。
- 覆盖索引：查询字段**全部在索引中**，无需回表，性能极高。

### 4. 联合索引最左前缀原则
联合索引 `(a,b,c)`
- 有效：a / a+b / a+b+c
- 失效：b、c、b+c
> 范围条件（> < like %）后面的索引全部失效

### 5. 索引失效常见场景
1. like 以 `%` 开头
2. 字段隐式类型转换（字符串不加引号）
3. or 两边字段未都建索引
4. not in、not exists、is not null
5. 联合索引不满足最左匹配
6. order by 字段无索引、函数操作索引字段

### 6. 哪些字段不建索引
频繁更新、数据量小、重复度极高（性别、状态）、大文本字段。

## 三、MVCC 底层
### 1. 什么是 MVCC
多版本并发控制，InnoDB实现**无锁读**，读写不阻塞，提升并发。

### 2. 核心组成
- undo log：保存数据历史版本
- ReadView：事务隔离快照
- 隐藏列：DB_TRX_ID、ROLL_PTR 版本链

### 3. RR 与 RC 区别
- RC：每次查询生成新 ReadView，会出现不可重复读。
- RR：事务首次查询生成一个 ReadView，复用，保证可重复读。

## 四、三大日志
### 1. redo log 重做日志
- 作用：**崩溃恢复**，保证事务持久性。
- 先写日志再写磁盘，宕机重启重做未落地数据。

### 2. undo log 回滚日志
- 作用：事务回滚、MVCC版本链。

### 3. binlog 二进制日志
- 作用：**主从复制、数据归档、恢复**。
- 三种格式：
    - statement：记录SQL，体积小，可能有函数时差。
    - row：记录行变更，精准，体积大（主流）。
    - mixed：混合模式。

## 五、锁机制
### 1. 锁分类
- 表锁：开销小、并发低，MyISAM常用。
- 行锁：InnoDB，粒度细、并发高，只锁当前行。
- 意向锁：表级辅助锁，快速判断表是否有行锁。

### 2. 行锁三种算法
- 记录锁：锁单行
- 间隙锁：锁间隙，防止插入
- 临键锁：记录锁+间隙锁，RR级别默认，解决幻读

### 3. 死锁
- 必要条件：互斥、请求保持、不可剥夺、循环等待。
- 解决：超时机制、死锁检测、统一加锁顺序、小事务。

### 4. 悲观锁 & 乐观锁
- 悲观锁：默认并发冲突多，for update 加锁。
- 乐观锁：版本号/时间戳，不加锁，适合读多写少。

## 六、SQL 优化
### 1. explain 关键字段
- type：性能核心，从优到劣：
  `system > const > eq_ref > ref > range > index > ALL`
- key：实际用到的索引
- Extra：
    - Using filesort：文件排序，需要优化
    - Using temporary：临时表，group by/join 问题

### 2. limit 大分页优化
原：`limit 100000,10`
优化：**主键回溯**
`select * from t where id > 100000 limit 10`

### 3. count 区别
- count(*)：统计行数，InnoDB最优，忽略null。
- count(1)：效果同count(*)。
- count(字段)：统计非null数据，效率最低。

### 4. 禁止 select *
多查无用字段、无法走覆盖索引、增加网络与IO开销。

## 七、主从复制 & 高可用
### 1. 主从复制流程
1. 主库写入记录binlog
2. 从库IO线程拉取binlog到本地relay-log
3. 从库SQL线程重放日志，同步数据

### 2. 主从延迟原因
网络延迟、主库大事务、从库单线程回放、大批量写入。

### 3. 分库分表
- 垂直拆分：按业务模块拆表
- 水平拆分：按id/时间分片拆大数据量表
  解决单表数据千万级性能瓶颈。

## 八、高频场景简答题
1. 大表加索引：使用 `ALGORITHM=INPLACE LOCK=NONE` 避免锁表。
2. 超卖解决：数据库悲观锁、乐观锁、分布式锁、库存原子扣减。
3. 长事务危害：锁久、undo log膨胀、库性能下降、主从延迟。
